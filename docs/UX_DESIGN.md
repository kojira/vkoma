## 1. ビジョンと設計思想

### vKomaが目指すユーザー体験

vKomaは「**スマホからAIにチャットで指示するだけで動画が作れるツール**」を目指す。ユーザーはコードを書かない。AIが自然言語の指示からシーンコードを自動生成し、タイムラインGUIで微調整して書き出す。

**理想のフロー:**
```
1. ユーザー: 「バンドのMVを作りたい。曲はこれ」（BGMをアップロード）
2. AI: 「どんな雰囲気ですか？ジャンルや色のイメージがあれば教えてください」
3. ユーザー: 「サイバーパンクっぽい感じで、紫と青のグラデーション」
4. AI: 「了解しました。こんな構成はどうですか？」→ プレビューが自動更新
5. ユーザー: 「タイトルをもう少し大きくして」
6. AI: パラメータを調整 → リアルタイム反映
7. ユーザー: スライダーで微調整 → 書き出し
```

### 「AIチャットファースト、GUIで微調整」の具体化

| レイヤー | 役割 | 例 |
|---|---|---|
| **AIチャット（主）** | 大きな構成決定、シーン生成、意図の伝達 | 「タイトルシーンを追加」「雰囲気を変えて」 |
| **パラメータパネル（副）** | 数値の微調整、色の変更 | フォントサイズを72→80に、色を#ffffffに |
| **タイムライン（副）** | 時間軸の調整、シーンの並び替え | シーンのdurationを3秒→4秒に |

**核心原則:**
- AIは**対話相手**であり、一方的な生成機ではない
- ユーザーの意図を理解してから提案する（理解確認→提案→承認→実行）
- 「何を作ろうとしているか」を常にユーザーに見せる
- 失敗した場合は何が問題かをユーザーに伝える
- **すべてのデータはサーバーが正（Single Source of Truth）**。フロントエンドはサーバーの状態を反映するビュー
- **保存ボタンは存在しない。すべての変更は即座にサーバーに自動保存される**
- **自動保存の代償としてUndo/Redoを提供する**。ユーザーは安心して変更を試行錯誤できる

### 現状との乖離

現在のvKomaは上記の理想から大きく乖離している：
- AIが質問せず一方的に生成する
- 生成結果の説明が不十分（「何を理解して何を作ろうとしているか」が見えない）
- チャット履歴がデバイス間で共有されない
- シーンが正常に動作しないケースが多い
- GUIでの微調整が困難（シーン選択不可、パラメータ変更不可）
- **データがフロントエンド（localStorage/Zustand）に分散保持されており、デバイス間同期もサーバー再起動耐性もない**
- **明示的な「保存」操作が必要で、保存し忘れによるデータ消失リスクがある**
- **Undo/Redoがなく、誤操作を取り消せない**

## 2. ユーザーフロー（ステップバイステップ）

### 2.1 新規プロジェクト作成〜MV完成の全工程

```
┌──────────────────────────────────────────────────────────────┐
│                     ユーザーフロー全体図                       │
│                                                              │
│  ① プロジェクト作成                                           │
│     └→ プロジェクト名入力 → テンプレート選択（任意）            │
│                                                              │
│  ② アセット準備                                               │
│     └→ BGM/画像/動画のアップロード                             │
│                                                              │
│  ③ AIとの対話（メインワークフロー）                             │
│     └→ 意図の伝達 → AIの理解確認 → 提案 → 承認/修正           │
│     └→ シーン生成 → プレビュー確認 → 修正指示 → 再生成         │
│     └→ 繰り返し（3〜10ターン程度で完成イメージに到達）          │
│                                                              │
│  ④ GUI微調整                                                  │
│     └→ タイムラインでシーン順序・duration調整                   │
│     └→ パラメータパネルでフォント・色・サイズ微調整              │
│     └→ トランジション設定                                      │
│                                                              │
│  ⑤ プレビュー・確認                                           │
│     └→ 全体通し再生 → 修正箇所の特定 → ③または④に戻る          │
│                                                              │
│  ⑥ 書き出し                                                   │
│     └→ MP4書き出し → ダウンロード                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 各ステップの詳細

#### ① プロジェクト作成

| ユーザーの操作 | システムの応答 |
|---|---|
| 「新規プロジェクト」ボタンをクリック | プロジェクト名入力ダイアログを表示 |
| プロジェクト名を入力（例：「IRIS OUT MV」） | プロジェクトを作成し、エディタ画面に遷移 |
| （任意）テンプレートを選択 | テンプレートのシーン構成を展開 |

**AIの初回メッセージ（プロジェクト作成直後）:**
```
こんにちは！新しいプロジェクト「IRIS OUT MV」を作成しました。

どんな動画を作りたいですか？
以下を教えてもらえると、より良いシーンを提案できます：
- 🎵 音楽のジャンル・テンポ
- 🎨 色やビジュアルのイメージ
- ⏱ おおよその動画の長さ
- 📝 表示したいテキストやロゴ

BGMがあれば、先にアップロードしてもらうと曲に合わせた構成を提案できます。
```

#### ② アセット準備

| ユーザーの操作 | AIの応答 |
|---|---|
| BGMファイルをアップロード | 「BGMを受け取りました。3分24秒の曲ですね。BPM分析中...」 |
| 画像をアップロード | 「ロゴ画像を受け取りました。120x60pxのPNG形式ですね。シーンに組み込めます」 |

**ポイント:** アセットアップロード時にAIが自動でメタデータを分析し、活用方法を提案する。

#### ③ AIとの対話（メインワークフロー）

**対話の原則:**

1. **理解確認フェーズ（1〜2ターン）**: AIがユーザーの意図を正しく理解しているか確認
2. **提案フェーズ（1ターン）**: AIが構成案を提案し、ユーザーの承認を求める
3. **実行フェーズ（1ターン）**: 承認後にシーンを生成
4. **微調整フェーズ（N ターン）**: ユーザーの修正指示に応じて調整

```
── 対話例 ──

ユーザー: 「バンドのMVっぽい動画を作りたい」

AI（理解確認）: 「バンドのMV風ですね！いくつか確認させてください：
1. テキストは曲名やバンド名を大きく表示する形でOKですか？
2. 背景はグラデーションや幾何学模様のどちらが好みですか？
3. 歌詞テロップは入れますか？」

ユーザー: 「テキストはバンド名"IRIS OUT"をメインで。背景はグラデーションで、
テロップはなしでいい」

AI（提案）: 「了解しました。こんな構成はどうでしょう？

📋 **提案する構成（全4シーン）:**

1. **イントロ（0:00〜0:04）**: 暗い紫→青のグラデーション背景に
   "IRIS OUT" がフェードイン
2. **メインA（0:04〜0:15）**: パーティクルエフェクト付き
   グラデーション背景、バンド名が中央に常駐
3. **メインB（0:15〜0:25）**: 色味を変えて（青→ピンク）
   ズームエフェクト付きテキスト
4. **アウトロ（0:25〜0:30）**: フェードアウトで終了

この方向で生成していいですか？修正があれば教えてください。」

ユーザー: 「いいね、それでお願い」

AI（実行）: 「生成しています...

✅ 4シーンを生成しました：
- イントロ: グラデーション背景（#1a1a2e→#16213e）に白テキスト、フェードイン
- メインA: パーティクル50個、紫背景、テキスト中央配置
- メインB: 青→ピンクグラデーション、ズームエフェクト
- アウトロ: 黒背景へのフェードアウト

プレビューで確認してみてください。気になる点があればいつでも言ってください。」
```

#### ④ GUI微調整

ユーザーがAIとの対話では伝えきれない細かい調整をGUIで行う：

- タイムライン上でシーンのdurationをドラッグで変更
- シーンをクリックしてパラメータパネルを開く
- スライダーでフォントサイズ・座標・透明度を調整
- カラーピッカーで色を変更
- ドロップダウンでエフェクト種類を切り替え

**すべての変更がリアルタイムでプレビューに反映される。**

#### ⑤ プレビュー・確認

- 再生ボタンで全体を通し再生
- タイムラインのシークバーで任意の位置にジャンプ
- 問題があればAIチャットまたはGUIで修正

#### ⑥ 書き出し

- 「書き出し」ボタンクリック → 進捗バー表示 → MP4ファイルダウンロード
- 書き出し中もプレビューは操作可能

### 2.3 AIが質問すべきタイミング

| タイミング | AIがすべきこと | 例 |
|---|---|---|
| **初回メッセージ** | 動画の目的・イメージを聞く | 「どんな動画を作りたいですか？」 |
| **曖昧な指示** | 具体化のための質問 | 「テキストのフォントは明朝体とゴシック体どちらが好みですか？」 |
| **大きな変更指示** | 影響範囲の確認 | 「全シーンの背景色を変えますか？それとも今選択中のシーンだけ？」 |
| **矛盾する指示** | 意図の確認 | 「先ほど青系と指定されましたが、暖色系に変更しますか？」 |
| **技術的制約** | 代替案の提案 | 「3D回転エフェクトは現在対応していませんが、ズーム＋回転で似た効果を出せます」 |

### 2.4 AIが質問すべきでないタイミング

| タイミング | AIがすべきこと |
|---|---|
| **明確な指示**（「フォントサイズを80にして」） | 即座に実行 |
| **軽微な修正**（「色をもう少し明るく」） | 適切に解釈して実行 |
| **削除・追加の指示**（「最後のシーンを消して」） | 即座に実行 |

## 3. AIチャットの設計

### 3.1 対話フロー設計

```
ユーザーのメッセージ
    │
    ▼
[意図分類]
    ├─ 情報提供のみ（雑談・質問） → テキスト応答のみ
    ├─ 明確な指示 → 即座にシーン生成/変更
    └─ 曖昧な指示 → 理解確認の質問
         │
         ▼
    [ユーザーの回答]
         │
         ▼
    [提案を提示]
    「こんな構成でどうですか？」
         │
         ├─ ユーザー承認 → シーン生成/変更を実行
         └─ ユーザー修正 → 提案を修正して再提示
              │
              ▼
         [シーン生成/変更]
              │
              ▼
         [結果の説明]
         「Xシーンを生成しました。
          各シーンの内容は...」
              │
              ▼
         [プレビュー自動更新]
```

### 3.2 AIの応答形式（改善版）

現状のAI応答はJSONのみだが、以下のように改善する：

**フェーズ1: ストリーミングテキスト表示**
```
AI: 「了解しました。サイバーパンク風のMVですね。
     紫と青のグラデーションをベースに、
     ネオン風のテキストエフェクトを使います...」
     ← ここまでリアルタイムにストリーミング表示
```

**フェーズ2: シーン生成（バックグラウンド）**
```
     [シーン生成中... ■■■□□]
     ← プログレスバーまたはスピナー表示
```

**フェーズ3: 結果の説明**
```
AI: 「✅ 3シーンを生成しました：
     1. タイトルシーン: ...
     2. メインシーン: ...
     3. アウトロ: ...
     プレビューで確認してみてください。」
```

### 3.3 チャット履歴のサーバー永続化

**現状の問題:**
- `chatStore.ts` は `zustand/persist` で `localStorage` に保存
- `ai-chat.ts` の `sessionHistories` は `Map` でインメモリ保持
- ブラウザリロード: クライアント側は復元されるがサーバーの会話コンテキストは消失
- 別デバイス: 両方とも消失

**設計原則: サーバーが正（Single Source of Truth）**

チャット履歴はフロントエンドに永続化しない。サーバーのファイルシステムが唯一の正のデータソースであり、フロントエンドはサーバーから取得した履歴を表示するだけのビューとなる。

```
┌─────────────────────────────────────────────────────┐
│ チャット履歴の保存先（サーバー側）                       │
│                                                     │
│  ~/vkoma-projects/<project-id>/chat-history.json     │
│                                                     │
│  {                                                  │
│    "sessions": [                                    │
│      {                                              │
│        "id": "session-xxx",                         │
│        "startedAt": "2026-03-17T12:00:00Z",         │
│        "messages": [                                │
│          { "role": "user", "content": "..." },      │
│          { "role": "assistant", "content": "..." }  │
│        ]                                            │
│      }                                              │
│    ],                                               │
│    "activeSessionId": "session-xxx"                  │
│  }                                                  │
│                                                     │
│  APIエンドポイント:                                    │
│  GET  /api/projects/:id/chat-history                 │
│  POST /api/projects/:id/chat-history                 │
│  DELETE /api/projects/:id/chat-history/:sessionId    │
└─────────────────────────────────────────────────────┘
```

**データフロー:**

```
ユーザーがメッセージ送信
    │
    ├─→ POST /api/ai/chat（メッセージ + コンテキスト）
    │     │
    │     ├─→ サーバー: chat-history.json にユーザーメッセージを追記保存（自動保存）
    │     ├─→ AI応答をストリーミング返却
    │     └─→ サーバー: chat-history.json にAI応答を追記保存（自動保存）
    │
    └─→ フロントエンド: 受信したメッセージをUIに表示（メモリ上のみ）

プロジェクトを開いた時:
    GET /api/projects/:id/chat-history → フロントエンドに履歴をロード

別デバイスで同じプロジェクトを開いた時:
    同じAPI → 同じ履歴が表示される
```

**クライアント側の変更:**
- `chatStore.ts` の `persist` (localStorage) を **完全に廃止**
- `chatStore` はインメモリ状態のみ持つ（サーバーから取得した履歴のキャッシュ）
- `loadProject()` 時にチャット履歴をサーバーからフェッチ
- メッセージ送受信は `/api/ai/chat` が自動的にサーバーに永続化するため、クライアント側の保存処理は不要

### 3.4 AIが参照すべき情報（コンテキスト）

AIが適切な応答をするためには、以下の情報をコンテキストとして参照する必要がある：

```
┌─────────────────────────────────────────────────────┐
│ AIへのコンテキスト注入                                  │
│                                                     │
│ 1. プロジェクト状態                                    │
│    - プロジェクト名                                    │
│    - 現在のシーン構成（名前・duration・パラメータ一覧）    │
│    - 総尺                                            │
│    - fps, 解像度                                     │
│                                                     │
│ 2. アセット情報                                       │
│    - アップロード済みアセット一覧（ID、名前、種類、尺）    │
│    - BGMの有無・BPM                                   │
│                                                     │
│ 3. タイムライン状態                                    │
│    - トラック構成                                      │
│    - 各トラックのアイテム配置                            │
│    - 現在選択中のシーン/トラックアイテム                  │
│                                                     │
│ 4. ユーザーの操作コンテキスト                            │
│    - 直前にユーザーがGUIで変更した内容                    │
│    - 現在表示中のプレビューフレーム                       │
│                                                     │
│ 5. 会話履歴（最新20メッセージ）                          │
└─────────────────────────────────────────────────────┘
```

**現状の問題:** `ai-chat.ts` ではプロジェクトIDとアセット情報しか渡していない。シーン構成やタイムライン状態は渡されていない。

**改善:** チャットリクエストに以下を追加：
```typescript
// ChatRequestBody の拡張
interface ChatRequestBody {
  sessionId: string;
  message: string;
  projectId: string;
  assets: ChatAsset[];
  // 追加
  context: {
    scenes: Array<{
      id: string;
      name: string;
      duration: number;
      params: Record<string, unknown>;
      sceneConfigId: string;
    }>;
    tracks: Array<{
      id: string;
      type: string;
      name: string;
      items: Array<{
        id: string;
        startTime: number;
        duration: number;
        sceneConfigId?: string;
        assetId?: string;
      }>;
    }>;
    selectedItemId: string | null;
    totalDuration: number;
    fps: number;
    width: number;
    height: number;
  };
}
```

### 3.5 ストリーミング表示の仕様

**現状の問題:**
- `ai-chat.ts` は Anthropic API のストリーミングを使用しているが、JSON全体をストリーミングしている
- クライアント側（`ChatPanel.tsx`）の `extractStreamingMessagePreview` がJSON内の `message` フィールドを部分的に抽出しようとするが、JSON構造が途中で崩れるため不安定

**改善案: 二段階ストリーミング**

```
Phase 1: メッセージテキストのストリーミング
  data: {"type":"text","content":"了解しました。"}
  data: {"type":"text","content":"サイバーパンク風の"}
  data: {"type":"text","content":"MVですね..."}

Phase 2: 構造化データの一括送信
  data: {"type":"thinking","content":"シーン構成を検討中..."}
  data: {"type":"scenes","scenes":[...]}
  data: {"type":"audioTracks","audioTracks":[...]}
  data: {"type":"message","content":"最終メッセージ（省略可）"}
  data: {"type":"done"}
```

**サーバー側の変更:**
- Anthropic API の `stream` を利用し、テキストチャンクを `type: "text"` で即座に送信
- JSON応答の解析は全テキスト受信後に行い、`type: "scenes"` / `type: "audioTracks"` で一括送信
- **`message` フィールドを JSON の先頭に配置する規約** は既に `ai-chat.ts` のシステムプロンプトに記載されているが、実際にはAI側が守らないケースがある。二段階ストリーミングで根本解決する

**クライアント側の変更:**
- `type: "text"` イベントを受信するたびにチャットバブルのテキストを追記表示（現在の `extractStreamingMessagePreview` を廃止）
- `type: "scenes"` 受信時にシーンを適用してプレビュー更新
- `type: "done"` 受信時に最終メッセージをフォーマット表示

## 4. タイムラインとマルチトラックの設計

### 4.1 現状の問題

| 問題 | 原因 | 影響 |
|---|---|---|
| マルチトラックが動かない | `timelineStore.ts` にTrack型定義はあるが、UIコンポーネントとPreviewCanvasが未対応 | オーバーレイ・テロップが作れない |
| 1トラック表示しかない | タイムラインUIが `sceneStore.scenes[]` を直列表示 | 複数トラックの同時編集不可 |
| シーン選択ができない | タイムラインでクリックしても `currentSceneIndex` が更新されない、または反映されない | 最初のシーンしかパラメータ表示されない |

### 4.2 トラック構成の設計

```
┌─────────────────────────────────────────────────────────────┐
│ タイムライン                                                  │
│                                                             │
│ トラック名    │  0s    2s    4s    6s    8s    10s            │
│ ─────────────┼────────────────────────────────────────────── │
```

## 5.5 サーバー中心のデータ管理と自動保存

### 設計原則: サーバーが唯一の正（Single Source of Truth）

**すべてのプロジェクトデータはサーバーのファイルシステムに保存される。** フロントエンドのZustandストアはサーバーから取得したデータの一時的なキャッシュ（ビュー）に過ぎず、永続化の責任を持たない。

```
┌─────────────────────────────────────────────────────┐
│ データの所在                                          │
│                                                     │
│  【サーバー（正）】                                    │
│  ~/vkoma-projects/<project-id>/                      │
│  ├── project.json       ← シーン構成・タイムライン     │
│  ├── chat-history.json  ← チャット履歴                │
│  ├── assets/            ← アセットファイル             │
│  ├── assets.json        ← アセットメタデータ           │
│  └── undo-history.json  ← 操作履歴（Undo/Redo用）     │
│                                                     │
│  【フロントエンド（キャッシュ）】                        │
│  Zustand stores (インメモリ)                          │
│  ├── timelineStore  ← サーバーの project.json を反映   │
│  ├── chatStore      ← サーバーの chat-history を反映   │
│  └── undoStore      ← サーバーの undo-history を反映   │
│                                                     │
│  localStorage / zustand/persist → 使用しない           │
└─────────────────────────────────────────────────────┘
```

**現状の問題:**
- `sceneStore.ts` と `timelineStore.ts` が `zustand/persist` で `localStorage` にデータを保存
- `chatStore.ts` も `zustand/persist` でローカル保存
- `saveProject()` を明示的に呼ばないとサーバーに反映されない
- デバイス間でデータが同期されない

**改善:**
- `zustand/persist` を全ストアから **廃止**
- すべての変更操作（パラメータ変更、シーン追加/削除、トラック操作）は即座にサーバーAPIを呼び出して保存
- プロジェクトを開いた時はサーバーからすべてのデータをフェッチ
- 「保存」ボタンは **存在しない**

### 自動保存の設計

**原則: ユーザーが意識せずにすべての変更が保存される。**

```
ユーザーの操作（パラメータ変更、シーン追加、ドラッグ等）
    │
    ├─→ Zustand store を即座に更新（UIに即反映）
    │
    └─→ 自動保存キュー（デバウンス: 300ms）
          │
          └─→ PATCH /api/projects/:id  ← 差分のみ送信
                │
                ├─→ サーバー: project.json を更新
                └─→ サーバー: undo-history.json に操作を記録
```

**デバウンスの理由:**
- スライダー操作時に毎フレーム（16ms）でAPIを叩くのは過剰
- 300msのデバウンスで、ユーザーがスライダーを動かし終わった時点で1回だけ保存
- ただしUIプレビューは即座に更新（サーバー保存を待たない）

**自動保存の実装方針:**

```typescript
// フロントエンド: 自動保存ミドルウェア
const autoSave = debounce(async (state: TimelineStore) => {
  if (!state.projectId) return;

  await fetch(`/api/projects/${state.projectId}`, {
    method: 'PATCH',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      tracks: state.tracks,
      // 差分のみ送信（将来的にはJSON Patchも検討）
    }),
  });
}, 300);

// Zustand の subscribe でストア変更を監視
timelineStore.subscribe((state, prevState) => {
  if (state.tracks !== prevState.tracks) {
    autoSave(state);
  }
});
```

**APIエンドポイントの変更:**

| メソッド | パス | 説明 | 変更点 |
|---|---|---|---|
| PATCH | `/api/projects/:id` | プロジェクトの部分更新（自動保存用） | **新規追加** |
| PUT | `/api/projects/:id` | プロジェクトの全体更新 | 既存（後方互換維持） |

**PATCH エンドポイントの設計:**
```typescript
// PATCH /api/projects/:id
// Body: 変更があったフィールドのみ
{
  "tracks": [...],         // トラック変更時
  "name": "...",           // 名前変更時
  // シーン追加/削除/パラメータ変更は tracks 内で表現
}
```

**保存状態のUI表示:**

```
┌──────────────────────────────────────────────┐
│ ヘッダーバー                                   │
│                                              │
│  vKoma │ IRIS OUT MV │ ✓ 保存済み             │
│                        ↑                      │
│                  保存状態インジケーター           │
│                  「✓ 保存済み」                  │
│                  「● 保存中...」                 │
│                  「⚠ 保存エラー（リトライ中）」   │
└──────────────────────────────────────────────┘
```

- 通常時: 何も表示しない（保存が完了している状態がデフォルト）
- 保存中: 小さなスピナーを表示
- エラー時: 警告アイコンを表示し、自動リトライ（3回まで、指数バックオフ）

### 5.6 Undo/Redo（操作履歴）

**設計原則: 自動保存があるからこそ、安心して元に戻せる機能が必須。**

保存ボタンがないため、ユーザーが誤った操作をした場合にすぐ元に戻せる必要がある。Undo/Redoは**サーバーとフロントエンドの両方で操作履歴を管理**する。

#### 操作履歴のデータモデル

```typescript
// 操作（Operation）の型定義
interface Operation {
  id: string;                    // 操作ID
  timestamp: string;             // ISO 8601
  type: OperationType;           // 操作の種類
  description: string;           // 人間向けの説明（例: "フォントサイズを64→80に変更"）
  // 復元用データ
  patch: JsonPatch;              // 適用する変更（JSON Patch形式）
  inversePatch: JsonPatch;       // 元に戻すための逆変更
}

type OperationType =
  | 'scene:add'
  | 'scene:remove'
  | 'scene:update-param'
  | 'scene:update-duration'
  | 'scene:reorder'
  | 'track:add'
  | 'track:remove'
  | 'track:reorder'
  | 'item:add'
  | 'item:remove'
  | 'item:move'
  | 'item:update-param'
  | 'transition:set'
  | 'project:rename'
  | 'ai:generate-scenes'       // AIによるシーン生成（まとめて1操作）
  | 'ai:modify-scenes';        // AIによるシーン修正

// Undo/Redo スタック
interface UndoHistory {
  operations: Operation[];       // 全操作履歴（古い順）
  cursor: number;               // 現在位置（undo/redoで移動）
  maxSize: number;              // 最大保持数（デフォルト: 100）
}
```

#### Undo/Redoのフロー

```
操作実行
    │
    ├─→ Zustand store 更新
    ├─→ 操作を Operation として記録
    │     ├─→ undoStore に追加（フロントエンド）
    │     └─→ POST /api/projects/:id/undo-history に記録（サーバー）
    └─→ 自動保存（project.json 更新）

Undo (Cmd+Z)
    │
    ├─→ undoStore.cursor を1つ戻す
    ├─→ operations[cursor].inversePatch をストアに適用
    ├─→ POST /api/projects/:id/undo（サーバー側もロールバック）
    └─→ PreviewCanvas 再描画

Redo (Cmd+Shift+Z)
    │
    ├─→ undoStore.cursor を1つ進める
    ├─→ operations[cursor].patch をストアに適用
    ├─→ POST /api/projects/:id/redo（サーバー側も再適用）
    └─→ PreviewCanvas 再描画
```

#### 操作のグルーピング

連続した細かい操作（スライダーのドラッグ中の各値変更など）を1つの操作としてグルーピングする：

```
スライダー操作開始（mousedown）
    │
    ├─→ 操作グループ開始: startGroup('scene:update-param')
    │     ├─→ 初期値を記録
    │
    ├─→ ドラッグ中の各値変更
    │     ├─→ UIに即反映（プレビュー更新）
    │     └─→ 操作履歴には記録しない
    │
    └─→ 操作完了（mouseup）
          ├─→ 操作グループ終了: endGroup()
          ├─→ 「フォントサイズを64→80に変更」として1つのOperationを記録
          └─→ 自動保存
```

#### AIによるシーン生成のUndo

AIが複数シーンを一括生成した場合、1回のUndoで全シーンをまとめて元に戻す：

```
AI: 4シーンを生成
    │
    └─→ Operation: {
           type: 'ai:generate-scenes',
           description: 'AIが4シーンを生成（イントロ、メインA、メインB、アウトロ）',
           patch: [4シーンの追加パッチ],
           inversePatch: [4シーンの削除パッチ],
         }

Cmd+Z:
    └─→ 4シーンがまとめて削除される（元の状態に戻る）
```

#### サーバー側API

| メソッド | パス | 説明 |
|---|---|---|
| GET | `/api/projects/:id/undo-history` | 操作履歴の取得 |
| POST | `/api/projects/:id/undo` | Undo実行（サーバー側のproject.jsonもロールバック） |
| POST | `/api/projects/:id/redo` | Redo実行（サーバー側のproject.jsonも再適用） |

#### キーボードショートカット

| ショートカット | 操作 |
|---|---|
| `Cmd+Z` / `Ctrl+Z` | Undo |
| `Cmd+Shift+Z` / `Ctrl+Shift+Z` | Redo |

#### UI表示

```
ヘッダーバー:
  [↩ Undo] [↪ Redo]    ← ボタン（グレーアウトで無効状態を表示）

ツールチップ:
  [↩ Undo] → 「フォントサイズを64→80に変更 を元に戻す」
  [↪ Redo] → 「フォントサイズを64→80に変更 をやり直す」
```

### 7.1 機能別ギャップ一覧

| 機能 | 現状 | あるべき姿 | ギャップ | 優先度 |
|---|---|---|---|---|

| **トランジション設定UI** | ロジックのみ（DESIGN_NEXT.md設計済み） | タイムラインUIから設定可能 | トランジション設定ダイアログ | **P2** |
| **アセットD&D** | アセットライブラリはある | タイムラインへのD&D | D&Dハンドラー実装 | **P2** |

```
timelineStore.tracks[] ←── 使われていない（UIとPreviewが未対応）
```

### B. 改善後のデータフロー（サーバーが正）

```
                    ┌──────────────────────────────────┐
                    │     サーバー（Single Source of     │
                    │          Truth）                  │
                    │                                  │
                    │  project.json       ← 自動保存   │
                    │  chat-history.json  ← 自動保存   │
                    │  undo-history.json  ← 自動保存   │
                    │  assets/            ← ファイル保存 │
                    └──────┬───────────────────────────┘
                           │
              ┌────────────┼────────────────┐
              ▼            ▼                ▼
    ChatPanel      timelineStore      undoStore
    (メモリ)       (メモリキャッシュ)   (メモリキャッシュ)
         │              │                │
         │              ▼                │
         │      PreviewCanvas            │
         │      (リアルタイム描画)         │
         │                               │
         └── 操作 ──→ ストア更新          │
                        │                │
                        ├─→ UI即反映      │

                        └─→ Operation記録 ──→ undoStore
                                               │
                                               └─→ POST /undo-history

Undo/Redo:
    Cmd+Z ──→ undoStore.undo()
               │
               ├─→ inversePatch をストアに適用
               ├─→ UI即反映
               └─→ POST /api/projects/:id/undo ──→ サーバーもロールバック
```

### C. パラメータ編集のデータフロー（自動保存+Undo対応）

```
タイムライン
    │
    ├─ シーンバークリック
    │     │
    │     └──→ timelineStore.selectedItemId = itemId
    │               │
    │               └──→ パラメータパネル
    │                       │
    │                       ├── sceneConfig.defaultParams からUIを自動生成
    │                       └── item.params から現在値をロード
    │
    └─ パラメータ変更（スライダー操作等）
          │
          ├─ mousedown: 操作グループ開始、初期値を記録
          │
          ├─ ドラッグ中: UIに即反映（プレビュー更新 16msデバウンス）
          │              → 自動保存はまだ発火しない
          │
          └─ mouseup: 操作グループ終了
               │
               ├──→ Operation を記録（「fontSize: 64→80」）
               ├──→ 自動保存発火（300msデバウンス）
               │      └──→ PATCH /api/projects/:id
               └──→ Undoスタックに追加
```
